SkyPulse — Full-Stack Weather Application

Status: Completed
Category: Web Application
Scope: Full-Stack REST Integration

A full-stack weather interface utilizing geolocation and public weather APIs to present immediate meteorological tracking indexes and weekly forecasts.

Project Overview & Objectives

Most weather apps either bury useful data behind ads or overwhelm the user with more numbers than they need. The objective behind SkyPulse was to build a clean, fast interface that surfaces exactly what matters, current conditions while handling the real-world unpredictability of geolocation permissions, API rate limits, and inconsistent network conditions gracefully.

The build was structured around three architectural priorities: a responsive frontend that feels instant even on slower connections, a backend layer that protects API keys and normalizes inconsistent data formats, and a UI that communicates atmospheric data visually rather than as a wall of text.

The Technical Pipeline

SkyPulse runs on a full-stack architecture split across three operational layers:

  • Geolocation Handling: On load, the app requests the user's coordinates via the browser's Geolocation API, with a manual city-search fallback for users who decline location access or are searching for a different city entirely.
  • REST API Integration: Coordinates are passed to a public weather API, which returns raw meteorological data — temperature, humidity, wind speed, precipitation probability, and multi-day forecasts — that the backend normalizes into a consistent response shape for the frontend to consume.
  • Dynamic Rendering: The frontend parses the normalized payload and renders current conditions, an hourly breakdown, and a 7-day forecast, with conditional UI states (loading, error, empty) handled explicitly rather than left to silently fail.

Key Engineering Decisions

A few decisions shaped how the app behaves under real-world conditions rather than just the happy path:

  1. Graceful Geolocation Fallback: Rather than treating denied location permissions as an error state, SkyPulse transitions smoothly into a manual search flow — the app stays useful instead of stalling on a permissions prompt.
  2. API Key Protection: Weather API requests are routed through a backend layer rather than called directly from the browser, keeping API credentials off the client and avoiding key exposure in network requests.
  3. Unit & Data Normalization: Temperature units, time zones, and forecast intervals are normalized server-side before reaching the frontend, so the UI layer never has to guess at formatting or handle inconsistent API response shapes.
